# TFT-Sniper 项目总结

## 项目定位

这是一个运行在 macOS 上的金铲铲辅助脚本，采用纯视觉方案：

- 不读内存
- 不注入进程
- 不依赖游戏内部接口
- 通过全局热键、屏幕截图、OCR 和鼠标点击完成识别与购买

当前实现以代码为准，早期设计文档 `plan.md` 已不再完全匹配现状。

## 当前核心流程

1. 监听 `Shift + D`
2. 定位金铲铲窗口 `com.tencent.jkchess`
3. 计算相对窗口的截图 ROI 和点击坐标
4. 等待商店刷新
   - `fixed`: 固定等待 `animation_delay`
   - `adaptive`: 先 `min_wait`，再用小探针区域轮询画面变化
5. 截取商店文字带
6. 使用 Apple Vision OCR 识别名称和价格
7. 按名字或价格命中规则筛选目标卡
8. 使用手工标定的 5 个卡槽点击中心执行批量购买

## 主要模块

- `main.py`: 主流程调度
- `trigger.py`: 全局热键监听，当前触发键为 `Shift + D`
- `window.py`: 金铲铲窗口定位、相对坐标解算
- `capture.py`: Quartz 截图、商店刷新等待、探针指纹
- `ocr.py`: Apple Vision OCR 与价格解析
- `matcher.py`: 名字/价格 OR 匹配
- `action.py`: 批量点击与拟人化点击时序
- `calibrate.py`: 一次性校准 ROI 和 5 个卡槽点击中心
- `config.toml`: 运行时配置，支持 `Cmd + Shift + R` 热重载

## 当前关键设计

### 1. 窗口相对坐标

所有校准坐标都可以保存为相对金铲铲窗口左上角的 Point 坐标。窗口移动时，脚本会自动跟随，不需要重新标定。

### 2. 手工点击点位

点击不再依赖 ROI 五等分推算，而是使用手工标定的 5 个点击中心点：

- 稳定性高于理论分槽中心
- 更适合固定布局的商店界面
- 多张命中时更不容易误点或漏点

### 3. 批量购买点击

多张卡需要购买时，当前实现不会每点一次就立刻移回原位，而是：

- 只记录一次鼠标原始位置
- 依次移动到每个目标点
- 保留短暂 settle / hold / gap 时间
- 点击完成后再统一移回原位

这主要用于提升 2~3 张连买时的注册稳定性。

### 4. 价格 OCR 规则

价格识别已做领域收敛：

- 合法价格固定为 `1~7`
- 对 OCR 文本先做 Unicode 归一化
- 从右往左提取最后一个合法价格数字

因此可正确处理常见噪声：

- `93 -> 3`
- `51 -> 1`
- `02 -> 2`
- `⑦ -> 7`

如果价格无法可靠识别，内部会记为未知值，不再显示负数金币。

## 当前推荐配置

### 稳定优先

适合当前机器和窗口大小比较固定的场景：

```toml
wait_mode = "fixed"
animation_delay = 0.40
debug = false
```

### 需要观察刷新时机时

```toml
wait_mode = "adaptive"
min_wait = 0.18
max_wait = 0.45
poll_interval = 0.02
debug = true
```

## 校准方式

运行：

```bash
.venv/bin/python calibrate.py
```

脚本会要求依次记录 7 个点：

1. 商店文字条左上角
2. 商店文字条右下角
3. 第 1 张卡点击中心
4. 第 2 张卡点击中心
5. 第 3 张卡点击中心
6. 第 4 张卡点击中心
7. 第 5 张卡点击中心

完成后会自动写入 `config.toml`。

## 已知局限

- OCR 对价格仍可能偶发缺失，但对金币图标导致的前缀噪声已经做了专门处理
- 纯视觉方案受游戏实际刷新动画影响，整体时延不可能低于动画稳定时间
- 如果窗口大小发生变化而不是仅移动，需要重新校准，或启用尺寸缩放逻辑

## 当前建议

- 日常使用优先 `fixed` 等待模式，逻辑最简单、开销最低
- 多张卡漏买时，优先调大 `[click]` 下的 `inter_click_ms` 和 `hold_ms`
- 调参阶段开 `debug = true`，稳定后关闭
